# Business Rules

## Идентификация

- `visibleId` — пользовательский номер оборудования.
- `visibleId` должен быть уникальным.
- `inventoryNumber` должен быть уникальным.
- Внутренний `equipment.id` не используется в API как публичный идентификатор.

## Структура оборудования

- Оборудование обязательно должно иметь:
  - название;
  - модель;
  - инвентарный номер;
  - участок;
  - ответственного сотрудника;
  - статус;
  - дату выдачи.
- Производитель не хранится в `Equipment` напрямую.
- Производитель определяется через `EquipmentModel -> Manufacturer`.
- Страна производства необязательна.

## Ответственный и сотрудники

- Ответственный сотрудник должен существовать.
- Ответственный сотрудник должен быть активным.
- Если сотрудник не найден или отключён, запись запрещается.

## Модель, участок и страна

- Модель должна существовать.
- Участок должен существовать.
- Страна должна существовать, если она указана.

## Статус

- Допустимы только значения enum:
  - `ACTIVE`
  - `RESERVE`
  - `REPAIR`
  - `MAINTENANCE`
  - `WRITTEN_OFF`
- Для UI используется человекочитаемая подпись статуса.

## Серийный номер

- `serialNumber` необязателен.
- Пустое значение преобразуется в `null`.
- Значение `б/н` не хранится в БД и также преобразуется в `null`.
- Отображение `б/н` — это ответственность UI, а не хранилища.

## Текстовые поля

Поля:

- `specifications`
- `operationText`
- `notes`

Правила:

- выполняется `trim()`;
- пустая строка превращается в `null`;
- максимальная длина — `4000` символов.

## Год выпуска

- `manufactureYear` необязателен.
- Если указан, должен быть целым числом в диапазоне `1900..2100`.

## Даты

### Общие правила

- `issueDate` обязательна.
- даты принимаются в формате `ДД.ММ.ГГГГ` на входе backend.

### Относительные ограничения

- год `commissioningDate` не может быть меньше `manufactureYear`;
- `issueDate` не может быть раньше `commissioningDate`;
- если `commissioningDate` не указана, `issueDate` не может быть раньше года выпуска.

## Аудит

- создание оборудования пишет полный срез значимых полей в аудит;
- обновление оборудования пишет только реально изменившиеся поля;
- история карточки строится по данным `audit_log`.

## Поиск

- после каждого создания и обновления оборудования обновляется поисковый индекс;
- индекс строится по имени, инвентарному номеру, модели, производителю, серийному номеру, местоположению и ответственному.

## Sequence видимого номера

- форма создания получает следующий безопасный `visibleId` через `IdentityNumberingService`;
- если пользователь вручную задаёт `visibleId` при создании или меняет его при обновлении, backend выполняет best-effort синхронизацию sequence;
- прямое изменение `visibleId` в БД считается вне штатного сценария.

## Транзакционность

- создание и обновление оборудования выполняются внутри одной транзакции;
- при обновлении строка оборудования блокируется `SELECT ... FOR UPDATE`;
- поиск и аудит обновляются в рамках той же транзакции записи.

## Файлы

- файлы оборудования хранятся через storage-модуль;
- оборудование выступает владельцем файлов;
- загрузка файла требует отдельного права управления файлами;
- загрузка файла пишет аудит в модуле `EQUIPMENT`.
