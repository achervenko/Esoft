# Architecture

## Общая структура

Модуль разделён на чтение данных, запись данных, получение истории изменений и работу с файлами.

HTTP-запросы обрабатываются двумя контроллерами:

- `EquipmentController` обслуживает основной REST API оборудования;
- `EquipmentFilesController` обслуживает файлы оборудования.

`EquipmentController` передаёт запросы фасаду `EquipmentService`. Фасад распределяет их между специализированными сервисами:

- `EquipmentQueryService` — чтение оборудования и справочников;
- `EquipmentHistoryService` — получение истории изменений;
- `EquipmentWriteService` — создание и обновление оборудования.

`EquipmentFilesController` использует `EquipmentService`, чтобы найти внутренний идентификатор оборудования по пользовательскому `visibleId`. Получение и загрузка файлов выполняются через `StorageFileService`.

Доступ к базе данных выполняется через `PrismaService`. Создание и обновление оборудования выполняются в транзакции, которая включает:

- запись оборудования;
- регистрацию аудита;
- обновление поискового индекса.

Файлы обрабатываются отдельно через storage-модуль и не входят в транзакции создания или обновления оборудования.

## EquipmentController

`EquipmentController` обслуживает основной REST API модуля:

- получение списка оборудования;
- получение справочников для формы;
- получение карточки оборудования;
- получение истории изменений;
- создание оборудования;
- обновление оборудования.

Контроллер:

- не содержит Prisma-логики;
- проверяет `visibleId` через `ParseIntPipe`;
- проверяет право на изменение через `assertCanEditEquipment()` перед созданием и обновлением;
- делегирует выполнение сценариев фасаду `EquipmentService`.

## EquipmentFilesController

`EquipmentFilesController` отвечает только за файлы оборудования:

- получение списка файлов;
- загрузку файла по `visibleId`.

Сначала контроллер вызывает `EquipmentService.findStorageOwnerByVisibleId()`, чтобы преобразовать пользовательский `visibleId` во внутренний `equipment.id`.

После определения владельца фактическая работа с файлами передаётся `StorageFileService`.

Перед загрузкой файла контроллер проверяет право пользователя через `assertCanManageFiles()`.

Файловые операции выполняются независимо от транзакций создания и обновления оборудования.

## EquipmentService

`EquipmentService` является фасадом модуля.

Его задача — предоставить контроллерам единый вход и распределить вызовы по специализированным сервисам:

- чтение списка, карточки и справочников — `EquipmentQueryService`;
- получение истории — `EquipmentHistoryService`;
- создание и обновление — `EquipmentWriteService`;
- определение владельца файлов — `EquipmentQueryService`.

Фасад не содержит собственной бизнес-логики и не обращается к Prisma напрямую.

## EquipmentQueryService

`EquipmentQueryService` отвечает за сценарии чтения данных.

### findAll()

Метод возвращает реестр оборудования.

Он:

- выбирает только поля, необходимые для таблицы;
- сортирует записи по `visibleId`;
- преобразует каждую запись в DTO через `toEquipmentListItem()`.

### getCreateOptions()

Метод возвращает справочные данные, необходимые для формы создания и редактирования:

- производителей;
- модели оборудования;
- страны;
- участки;
- активных сотрудников;
- допустимые статусы;
- следующий видимый номер `nextVisibleId`.

Для получения `nextVisibleId` используется `IdentityNumberingService.getNextId()`.

### findOneByVisibleId()

Метод загружает карточку оборудования по пользовательскому `visibleId`, а не по внутреннему `equipment.id`.

Полученная запись преобразуется в DTO через `toEquipmentCard()`.

Если оборудование не найдено, метод выбрасывает `NotFoundException`.

### findStorageOwnerByVisibleId()

Метод определяет владельца файлов по пользовательскому `visibleId`.

Он возвращает данные, используемые storage-модулем:

- `entityId` — внутренний `equipment.id`;
- `entityType` — `"equipment"`;
- `module` — `StorageOwnerModule.EQUIPMENT`.

Если оборудование не найдено, файловая операция не выполняется.

## EquipmentHistoryService

`EquipmentHistoryService` отвечает за получение истории изменений оборудования.

Последовательность работы:

1. Найти оборудование по пользовательскому `visibleId`.
2. Получить внутренний `equipment.id`.
3. Прочитать записи `audit_log`, относящиеся к найденному оборудованию.
4. Ограничить выборку значениями:
   - `entityId = equipment.id`;
   - `entityType = "equipment"`;
   - `module = AuditModule.EQUIPMENT`.
5. Подтянуть сведения о пользователе и связанном сотруднике.
6. Сформировать отображаемое имя автора через `toAuditUserName()`.
7. Вернуть записи в обратном хронологическом порядке.

Если оборудование не найдено, сервис выбрасывает `NotFoundException`.

## EquipmentWriteService

`EquipmentWriteService` является основным координатором операций записи.

Он отвечает за:

- подготовку входных данных;
- открытие транзакции;
- проверку ссылочных сущностей;
- создание или обновление оборудования;
- регистрацию аудита;
- обновление поискового индекса;
- получение итоговой карточки;
- синхронизацию sequence видимого номера.

Низкоуровневая проверка и нормализация пользовательского payload вынесены в `buildEquipmentData()`.

Проверка существования модели, участка, страны и ответственного сотрудника вынесена в `EquipmentReferenceValidatorService`.

### Последовательность create()

Создание оборудования выполняется следующим образом:

1. `CreateEquipmentDto` передаётся в `buildEquipmentData()`.
2. Входные данные проверяются и нормализуются.
3. Переданный пользователем `visibleId` отдельно обрабатывается через `parseEquipmentVisibleId()`.
4. Если `visibleId` указан вручную, предварительно проверяется, что он не занят.
5. Открывается Prisma-транзакция.
6. Внутри транзакции проверяются ссылочные сущности.
7. Создаётся запись оборудования.
8. Обновляется запись оборудования в `search_index`.
9. Через `AuditLogService.logEquipmentCreated()` регистрируется создание оборудования.
10. Созданная запись повторно загружается со всеми связями, необходимыми для карточки.
11. Результат преобразуется через `toEquipmentCard()`.
12. После успешной транзакции при ручном `visibleId` выполняется синхронизация sequence.

Ошибка синхронизации sequence не отменяет уже выполненное создание оборудования.

### Последовательность update()

Обновление оборудования выполняется следующим образом:

1. `CreateEquipmentDto` передаётся в `buildEquipmentData()`.
2. Входные данные проверяются и нормализуются.
3. Новый `visibleId`, если он передан, отдельно обрабатывается через `parseEquipmentVisibleId()`.
4. Открывается Prisma-транзакция.
5. Внутри транзакции проверяются ссылочные сущности из новых данных.
6. По текущему `visibleId` выполняется блокирующий запрос `SELECT ... FOR UPDATE`.
7. Если оборудование не найдено, выбрасывается `NotFoundException`.
8. Загружается текущее состояние оборудования.
9. Если `visibleId` изменяется, проверяется уникальность нового значения.
10. Запись оборудования обновляется.
11. Обновляется поисковый индекс.
12. Обновлённая запись повторно загружается со всеми необходимыми связями.
13. Через `AuditLogService.logEquipmentUpdated()` записываются фактические изменения.
14. Результат преобразуется через `toEquipmentCard()`.
15. После успешной транзакции при изменении `visibleId` выполняется синхронизация sequence.

Блокировка строки предотвращает одновременное неконтролируемое обновление одной записи оборудования.

### runEquipmentWrite()

`runEquipmentWrite()` централизует выполнение операций записи.

Метод:

- открывает Prisma-транзакцию;
- выполняет переданную операцию;
- перехватывает уникальные конфликты Prisma;
- передаёт их в `throwIfEquipmentUniqueConflict()`.

Функции из `equipment-write.errors.ts` преобразуют ошибки Prisma `P2002` в понятные бизнес-ошибки для:

- занятого `visibleId`;
- занятого `inventoryNumber`.

Неизвестные ошибки не маскируются и передаются выше без изменения.

## EquipmentReferenceValidatorService

`EquipmentReferenceValidatorService` выполняет ссылочную валидацию данных оборудования.

Сервис проверяет:

- существование модели оборудования;
- существование участка;
- существование страны, если она указана;
- существование ответственного сотрудника;
- активность ответственного сотрудника.

Проверки выполняются внутри транзакции записи. Это снижает вероятность изменения связанных данных между проверкой и сохранением оборудования.

## buildEquipmentData()

Функция `buildEquipmentData()` из `equipment-data.mapper.ts` отвечает за проверку и подготовку пользовательского payload.

Она:

- проверяет обязательные поля;
- удаляет пробелы по краям строк через `trim()`;
- преобразует пустые необязательные строки в `null`;
- преобразует значение `serialNumber = "б/н"` в `null`;
- проверяет максимальную длину текстовых полей;
- проверяет числовые идентификаторы;
- проверяет допустимость статуса;
- преобразует строки дат в объекты `Date`;
- проверяет корректность календарных дат;
- проверяет взаимный порядок дат;
- проверяет диапазон года выпуска;
- подготавливает объект данных для Prisma.

Проверка `visibleId` выполняется отдельно через `parseEquipmentVisibleId()`.

## EquipmentSearchProjector

После создания и обновления оборудования `EquipmentWriteService` вызывает `EquipmentSearchProjector.upsertEquipment()`.

Проектор:

- читает минимально необходимый набор данных оборудования;
- собирает `title`, `subtitle` и `searchText`;
- записывает или обновляет запись в `search_index`.

Таким образом, глобальный поиск всегда отражает актуальное состояние оборудования, хотя сам пользовательский поиск относится к соседнему search-модулю.

## AuditLogService

`AuditLogService` отвечает за создание строк аудита.

Для модуля `Equipment` используются два сценария:

- `logEquipmentCreated()` — записывает значимые поля новой записи;
- `logEquipmentUpdated()` — пишет только реальные изменения между старым и новым состоянием.

Сами различия по полям вычисляются через `getEquipmentAuditChanges()`.

## IdentityNumberingService

Используется в двух местах:

- `EquipmentQueryService.getCreateOptions()` — получить следующий безопасный `visibleId` для формы;
- `EquipmentWriteService` — синхронизировать sequence после ручного `visibleId` на create или update.

Синхронизация выполняется в режиме best-effort: ошибка синхронизации логируется, но не откатывает успешную запись оборудования.
