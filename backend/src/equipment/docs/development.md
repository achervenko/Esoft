# Development Guide

## Общие правила

- Не писать Prisma-запросы из контроллеров.
- Не добавлять бизнес-логику в `EquipmentController`.
- Считать `EquipmentService` фасадом, а не местом для логики.

## Чтение данных

- Все read-only сценарии добавлять в `EquipmentQueryService`.
- Историю изменений добавлять и поддерживать через `EquipmentHistoryService`.
- Для списков использовать минимальный `select`, а не широкие `include`.

## Изменение данных

- Все изменения оборудования выполнять через `EquipmentWriteService`.
- Не обходить `buildEquipmentData()` при создании и обновлении.
- Все записи выполнять внутри транзакции.
- При обновлении сохранять блокировку строки оборудования через `SELECT ... FOR UPDATE`.

## Ссылочные проверки

- Проверки существования модели, участка, страны и ответственного выполнять через `EquipmentReferenceValidatorService`.
- Не дублировать эти проверки в контроллере или presenter-слое.

## Конфликты и ошибки

- Prisma-конфликты уникальности преобразовывать через `equipment-write.errors.ts`.
- Не отдавать raw Prisma errors напрямую в API.

## Аудит и поиск

- Любое изменение оборудования должно писать аудит.
- Любое изменение оборудования должно обновлять поисковый индекс через `EquipmentSearchProjector`.
- Если добавляется новое поле оборудования, оценить:
  - должно ли оно участвовать в аудите;
  - должно ли оно участвовать в поиске;
  - должно ли оно отображаться в истории или карточке.

## Sequence и visibleId

- Новый пользовательский `visibleId` для формы получать через `IdentityNumberingService`.
- После ручного `visibleId` на create/update синхронизировать sequence через существующую best-effort логику.
- Не полагаться на прямые правки в БД как на штатный сценарий.

## Presenter и DTO

- Не возвращать raw Prisma entities из публичного API.
- Для списка использовать `toEquipmentListItem()`.
- Для карточки использовать `toEquipmentCard()`.
- Справочники формы формировать через `toEquipmentCreateOptions()`.

## Изменение схемы данных

При добавлении нового поля оборудования нужно проверить всю цепочку:

```text
Prisma schema
→ миграция БД
→ CreateEquipmentDto
→ buildEquipmentData()
→ EquipmentWriteService
→ presenter
→ audit
→ search projector (если нужно)
→ frontend types
→ create/edit form
→ equipment card / registry
```

## Файлы

- Не привязывать загрузку файлов напрямую к внутреннему `equipment.id` на уровне внешнего API.
- Для файлов всегда использовать `visibleId` в URL и переводить его во внутреннего владельца через `findStorageOwnerByVisibleId()`.

## Поддержка документации

- Документация в `README.md` и `docs/` должна обновляться вместе с изменениями модуля.
- Документы должны описывать текущую реализацию, а не желаемое будущее состояние.
