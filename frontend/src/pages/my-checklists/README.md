# Мои чек-листы

Документ описывает пользовательский frontend-сценарий выполнения рабочих чек-листов.

Экран предназначен для:

- просмотра назначенных чек-листов;
- старта собственного чек-листа;
- промежуточного сохранения ответов;
- завершения чек-листа;
- повторного открытия чек-листа в работе;
- просмотра завершённых чек-листов.

Административный каталог шаблонов и событий оборудования описан отдельно:

- `modules/checklists/README.md`
- `modules/equipment-events/README.md`

## Route и доступ

Экран открыт по route:

```text
#/my-checklists
```

Вкладки переключаются query-параметром:

```text
#/my-checklists?tab=new
#/my-checklists?tab=in-progress
#/my-checklists?tab=completed
```

Пункт `Мои чек-листы` виден всем ролям.

Экран отображает только чек-листы, назначенные текущему пользователю. Backend выполняет фильтрацию по `assignedUserId` на основе текущей авторизации и для списка, и для detail. Frontend дополнительно ограничивает доступные действия по текущему пользователю через `assignedUser.id === currentUserId`.

## Основная модель

Пользователь работает не с шаблоном, а с экземпляром чек-листа, уже назначенным в событии оборудования.

Экземпляр содержит:

- шаблон;
- оборудование;
- вид обслуживания;
- плановую дату события;
- статус;
- прогресс;
- набор модулей и вопросов;
- `version` для optimistic locking.

Статусы рабочих чек-листов:

```ts
type ChecklistWorkStatus =
  | "CREATED"
  | "IN_PROGRESS"
  | "COMPLETED"
  | "CANCELLED"
  | "INVALIDATED";
```

Основной пользовательский UX первого релиза использует три вкладки:

- `Новые` -> `CREATED`;
- `В работе` -> `IN_PROGRESS`;
- `Завершённые` -> `COMPLETED`.

`CANCELLED` и `INVALIDATED` не выводятся в основные вкладки.

## Структура страницы

Основная точка входа:

- `MyChecklistsPage.tsx`

Разделение по файлам:

- `hooks/use-my-checklists-list.ts` — загрузка списка активной вкладки;
- `hooks/use-checklist-detail.ts` — выбранная карточка и защита от гонок detail-запросов;
- `hooks/use-checklist-draft.ts` — draft-ответы и changed answers;
- `hooks/use-checklist-actions.ts` — start/save/complete;
- `hooks/use-checklist-work.ts` — композиция этих хуков;
- `components/` — список, detail и поля ответов;
- `my-checklists.answers.ts` — чистая логика преобразования ответов;
- `my-checklists.errors.ts` — преобразование API-ошибок в сообщения.

## Список чек-листов

Список показывает:

- название шаблона;
- оборудование;
- модель;
- вид обслуживания;
- плановую дату;
- статус;
- прогресс.

На карточке списка доступны действия:

- `Начать` для `CREATED`, если чек-лист назначен текущему пользователю;
- `Продолжить` / `Открыть` через открытие detail;
- `Просмотреть` для read-only сценариев.

После успешного старта из вкладки `Новые` route автоматически переключается на:

```text
#/my-checklists?tab=in-progress
```

Это нужно, чтобы запущенный чек-лист не исчезал из списка текущей вкладки и карточка не закрывалась сразу после старта.

## Detail-панель

Правая панель показывает:

- название шаблона;
- исполнителя;
- вид обслуживания;
- summary по оборудованию и датам;
- список модулей и вопросов;
- кнопки действий.

Для завершённого чек-листа экран становится режимом только чтения.

## Вопросы и draft-ответы

Поддерживаются типы:

- `BOOLEAN`
- `INTEGER`
- `DECIMAL`
- `TEXT`
- `DATE`

Draft-хранилище во frontend:

```ts
type DraftAnswers = Record<number, string>;
```

Frontend хранит черновые значения строками и переводит их в API-формат только перед отправкой.

Для обязательных вопросов проверка выполняется до завершения чек-листа. Для `BOOLEAN` значением считается только явный выбор `true` или `false`.

## API

Экран использует:

- `GET /api/checklists`
- `GET /api/checklists/:id`
- `POST /api/checklists/:id/start`
- `PATCH /api/checklists/:id/answers`
- `POST /api/checklists/:id/complete`

Список поддерживает query-параметры:

- `status`
- `limit`
- `offset`
- `eventId`
- `equipmentVisibleId`
- `dateFrom`
- `dateTo`

## Start / Save / Complete

### Старт

`startChecklistWork(...)`:

- переводит текущий чек-лист в `IN_PROGRESS`;
- может автоматически перевести событие в `IN_PROGRESS` на backend;
- после успеха frontend переключает вкладку на `В работе`.

### Сохранение ответов

`saveChecklistWorkAnswers(...)`:

- не меняет статус события на frontend;
- отправляет только изменённые ответы;
- после успеха перечитывает detail чек-листа с backend, чтобы получить актуальные `answeredAt`, `progress` и `version`;
- отдельно пытается обновить список вкладки.

Если обновление detail после сохранения не удалось, сохранение не считается достаточно актуализированным для последующего завершения тем же действием.

### Завершение

`completeChecklistWork(...)`:

- запрашивает подтверждение до любых дополнительных backend-изменений;
- при наличии несохранённых ответов сначала вызывает сохранение;
- продолжает завершение только если после сохранения удалось получить актуальную карточку с новой `version`;
- после успеха может получить от backend уже автоматически завершённое событие.

## Ошибки и сообщения

Внутри page разделены несколько состояний:

- `listError` — ошибка загрузки списка;
- `detailError` — ошибка открытия или перечитки карточки;
- `formError` — ошибка действий пользователя или backend-валидации;
- `versionConflict` — конфликт optimistic locking;
- `refreshError` — операция выполнена, но список/карточка не удалось синхронно обновить;
- `message` — успешное пользовательское сообщение.

`CHECKLIST_VERSION_CONFLICT` показывается отдельно и даёт пользователю явно перезагрузить чек-лист.

## Защита от гонок

Во frontend есть две защиты от конкурирующих запросов:

- `use-my-checklists-list.ts` инвалидирует старые reload списка через `requestIdRef`;
- `use-checklist-detail.ts` защищает detail-запросы через `detailRequestIdRef`.

Это нужно, чтобы:

- выбор нового чек-листа не перетирался более медленным старым запросом;
- reload после смены вкладки не записывал устаревший список.

## Ограничения текущей версии

- отдельной вкладки для `CANCELLED` и `INVALIDATED` нет;
- offline-draft и локальное кеширование между сессиями не реализованы;
- глобальный пользовательский экран фильтрации по оборудованию или датам поверх `#/my-checklists` пока не добавлен;
- автоматические frontend-тесты для этого экрана отсутствуют.
