# События оборудования

Документ описывает frontend-модуль событий оборудования:

- список событий в карточке оборудования;
- создание и редактирование события в статусе `CREATED`;
- просмотр карточки события;
- отмену события;
- подготовку данных формы назначения чек-листов по ответственным.

Пользовательский экран выполнения чек-листов описан отдельно в `pages/my-checklists/README.md`.

## Где находится модуль

Основная точка входа:

- `EquipmentEventsPanel.tsx`

События встраиваются во вкладку `События` карточки оборудования и не имеют собственного route верхнего уровня.

## Права и доступ

Во frontend есть два уровня работы с модулем:

- просмотр списка и карточки события;
- управление событием.

Флаг `canManageEvents` приходит в `EquipmentEventsPanel` извне. При `false` пользователь всё ещё может видеть список и карточку события, но не получает сценарии создания, редактирования и отмены.

Управляющий сценарий рассчитан на:

- `admin`;
- `chief_engineer`.

Внутри панели используются два связанных флага:

- `canManageEvents` — внешний признак, что пользователю вообще доступны управляющие сценарии модуля;
- `canEditEvents` — производный флаг для формы события. Он становится `true`, только если:
  - `canManageEvents === true`;
  - справочные данные формы успешно загружены;
  - нет ошибки загрузки form data;
  - в списке доступных ответственных есть хотя бы один пользователь.

Из-за этого пользователь может иметь право управления модулем в целом, но временно не получить кнопку редактирования, если frontend не смог подготовить данные формы.

## Что поддерживается

Поддерживаются:

- просмотр списка событий конкретного оборудования;
- просмотр detail-модали события;
- создание ручного события;
- редактирование события в статусе `CREATED`;
- отмена события в статусах `CREATED` и `IN_PROGRESS`;
- отображение прогресса по каждому назначенному чек-листу;
- назначение одного шаблона чек-листа каждому ответственному.

Не поддерживаются:

- ручной старт события;
- ручное завершение события;
- отдельная модаль `factDate`.

Событие переходит в `IN_PROGRESS` автоматически при старте первого рабочего чек-листа, а в `COMPLETED` автоматически при завершении последнего.

## Структура UI

Основные файлы:

- `EquipmentEventsPanel.tsx` — контейнер панели;
- `useEquipmentEventsPanel.ts` — координация списка, detail, form data и мутаций;
- `EquipmentEventsTable.tsx` — таблица списка;
- `EquipmentEventsPanelModals.tsx` — модали формы, отмены и просмотра;
- `EquipmentEventFormModal.tsx` — форма create/edit;
- `equipment-event-form/` — разнесённая логика формы.

## Список событий

Таблица показывает:

- вид обслуживания;
- статус;
- плановую дату;
- фактическую дату;
- тип выполнения;
- ответственных;
- агрегат по чек-листам вида `N из M завершено`.

Действия в таблице:

- `Открыть` — доступно всегда;
- `Изменить` — только при `canEditEvents` и статусе `CREATED`;
- `Отменить` — только при `canManageEvents` и статусах `CREATED` или `IN_PROGRESS`.

Кнопок `Взять в работу` и `Завершить` для события во frontend больше нет.

## Карточка события

Detail-модаль события показывает:

- базовые поля события;
- оборудование и производителя модели;
- список responsibles;
- список связанных чек-листов;
- для каждого чек-листа:
  - исполнителя;
  - название шаблона;
  - статус;
  - прогресс `answered/total`.

Карточка использует backend detail события, а не дополнительные клиентские дозагрузки шаблонов.

## Форма создания и редактирования

Форма остаётся модальной и используется в двух режимах:

- `create`;
- `edit`.

Основные поля:

- `maintenanceTypeId`;
- `plannedDate`;
- `responsibleUserIds`;
- назначения шаблонов чек-листов по ответственным;
- `note`.

Минимум один ответственный обязателен. Для каждого выбранного ответственного обязателен ровно один шаблон чек-листа.

## Модель назначений чек-листов

Форма работает по модели:

- один ответственный -> один чек-лист;
- один и тот же шаблон можно назначить нескольким ответственным;
- массив `checklistAssignments` описывает итоговое состояние назначений.

Структура assignment:

```ts
type EquipmentEventChecklistAssignment = {
  assignedUserId: string;
  checklistTemplateId: number;
};
```

## Роль maintenance settings

Настройка обслуживания больше не хранит массив шаблонов для события. Во frontend она используется только как источник default-шаблона для предзаполнения формы.

Правила:

- `defaultChecklistTemplate` подставляется каждому выбранному ответственному, если он существует и имеет `state === "ACTIVE"`;
- если default отсутствует или архивирован, форма не блокируется;
- пользователь должен выбрать шаблон вручную для каждого ответственного;
- backend-валидация события не зависит от наличия default в настройке.

## Логика формы

Разделение по файлам:

- `use-equipment-event-options.ts` — подготовка maintenance settings и responsible options;
- `use-equipment-event-assignments.ts` — изменение ответственных и их шаблонов;
- `equipment-event-form-submit.ts` — чистая подготовка payload;
- `use-equipment-event-form.ts` — координация состояния формы.

При смене вида обслуживания:

- пересчитывается active default для нового `maintenanceTypeId`;
- выбранным ответственным подставляется новый default, если он доступен;
- уже выбранное значение сохраняется, если нового default нет.

При добавлении ответственного:

- ему подставляется текущий active default;
- если default нет, шаблон остаётся пустым до ручного выбора.

## Отправка payload

Create отправляет:

```ts
{
  maintenanceTypeId,
  plannedDate,
  note,
  responsibleUserIds,
  checklistAssignments
}
```

Edit отправляет тот же полный набор в локальную модель формы, а для backend дополнительно строит `updatePayload` с `version` и только изменёнными полями.

`checklistAssignments` добавляется в `updatePayload`, если:

- изменился состав множества `responsibleUserIds`;
- изменился `maintenanceTypeId`;
- изменился `equipmentVisibleId`;
- изменилось хотя бы одно назначение шаблона у существующего ответственного.

## Загрузка справочных данных формы

`useEquipmentEventFormData.ts` загружает:

- maintenance settings;
- активные checklist templates;
- список доступных ответственных пользователей.

Шаблоны чек-листов загружаются всегда не для detail-модали события, а для формы create/edit. Detail уже получает названия шаблонов и прогресс чек-листов непосредственно из backend ответа события.

Хук защищён от гонок reload через `latestReloadIdRef` и не очищает предыдущие данные до успешного ответа.

## API, используемые во frontend

Используются:

- `GET /api/equipment-events`
- `GET /api/equipment-events/:id`
- `POST /api/equipment/:visibleId/events/manual`
- `PATCH /api/equipment-events/:id`
- `POST /api/equipment-events/:id/cancel`
- `GET /api/equipment-events/responsible-users`

Во frontend событий больше не используются:

- `POST /api/equipment-events/:id/start`
- `POST /api/equipment-events/:id/complete`

## Важные ограничения

- frontend не хранит собственный прогресс события; он показывает агрегаты по массиву `checklists[]`, пришедшему из API;
- archive-шаблон можно увидеть в уже созданном событии, но выбрать его заново в форме нельзя;
- отдельный экран общего администрирования событий вне карточки оборудования в этом модуле отсутствует.
